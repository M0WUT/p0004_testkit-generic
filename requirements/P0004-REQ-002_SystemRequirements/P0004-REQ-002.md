Requirement for the generic test equipment module
# Mechanical
| Requirement ID | Requirement | Justification |
| --- | --- | --- |
| MECH-01 | Generic Test Equipment (GTE) must use a COTS enclosure. (Use of PCB fabrication techniques to make custom panels is permitted) | Enclosure must be able to be externally sourced in quantity without the cost of custom mechanical parts |
| MECH-02 | GTE must allow for custom hardware to be installed in the enclosure. (Ideally a PCB as close to, but not exceeding, 100mm x 100mm with access to an external panel for connectors) | There is a standardised mechnical space allocated for custom hardware on a per-design basis. |


# Connectivity
| Requirement ID | Requirement | Justification |
| --- | --- | --- |
| CONN-01 | The GTE must have an Ethernet connection on a standard 8P8C connector (any speed, even 10BASE-T, is acceptable) | So the test equipment can be communicated with over Ethernet |
| CONN-02 | The GTE must present a Pyvisa compatible interface on its Ethernet connector | To ease integration with existing M0WUT testing codebase |
| CONN-03 | The GTE must support both static and DHCP IP address allocation | To allow operation in environments with and without DHCP servers | 
| CONN-04 | The GTE must not rely on an Internet connection for any operation. (It is acceptable to have an Internet-dependant way of performing a task if it makes it simpler or better experience but there must be an offline alternative without a reduction in functionality) | Devices should not be crippled because you don't want them browsing the Web.
| CONN-05 | The GTE should provide a web interface that can be accessed using a browser to allow for basic device configuration | Allows for reconfiguration without requiring specialist tools / software | 
| CONN-06 | The GTE must have a USB connection on USB Type-C (any generation of USB is acceptable) | To allow for firmware updating and communcations. USB Type-C because nothing else is acceptable in 2026! (real reason is to support USB-PD) |
| CONN-07 | The GTE must present a Pyvisa compatible interface on its USB connector | To ease integration with existing M0WUT testing codebase |
| CONN-08 | The GTE must be firmware upgradeable over either (or both) its Ethernet or USB connection without specialist hardware. This must not require the unit to be disassembled. | End users should not have to disassemble the unit to perform a firmware upgrade |
| CONN-09 | The GTE should have support for WiFi connections. It is acceptable if this requires a separate module to be retro-fitted to upgrade to this | Some testkit may benefit from wireless connectivity. This is expected to be a niche use case. |
| CONN-10 | The GTE must present a Pyvisa compatible interface over WiFi (if fitted with appropriate hardware) | To ease integration with existing M0WUT testing codebase |
| CONN-11 | The GTE need only support one Pyvisa connection at any given time | Only one external piece of software should be controlling the insturment at any given time |
| CONN-12 | The GTE should support exporting data to a Prometheus server | Allows for long-term logging and analysis on external devices

# Power
| Requirement ID | Requirement | Justification |
| --- | --- | --- |
| POW-01 | The GTE must support being powered by a USB-PD 2.0 (or later) PSE using the same connector as detailed in the "Connectivity" section. (It it acceptable to not boot until a higher voltage than the default 5V has been supplied) | To minimise likelihood that a specialist power supply is needed | 
| POW-02 | The GTE must support being powered from an 802.3 compliant PSE (e.g. 802.3af / 802.3at etc). As this is expected to be a niche use case, it is acceptable that an extra hardware module be required to support PoE operation. | May be very useful for remote sensing applications. As the cost of an insolated PoE DCDC is likely high, it should be an optional module to keep base unit cost down. |
| POW-03 | The GTE must support being powered from either a 2.1mm barrel jack or Anderson Powerpoles | To ensure compatability with common power connectors found in ham shacks | 
| POW-04 | The GTE should support being powered from both a 2.1mm barrel jack and Anderson Powerpoles | More options, more better
| POW-05 | The GTE shall feature reverse-polarity protection on both the 2.1mm barrel jack and Anderson Powerpoles (all that are fitted) | Cables for these interfaces are likely hand-assembled and risk having errors. USB and PoE are deemed to be sufficiently low risk that protection is not necessary as USB is unlikely to be hand-assembled and PoE is polarity agnostic |
| POW-06 | The GTE should only draw significant power (>0.1W) from one of its power options. It must be safe to plug in all possible power supplies without damaging either the power supplies or the GTE. | Power supplies should be properly multiplexed and only one source should be used at a time | 
| POW-07 | The GTE should prioritise which power supply to use in the following order: <ul><li>2.1mm barrel jack / Anderson Powerpole (if both connectors supported, priority can be either way round)</li><li>USB-PD (if negotitated correctly)</li><li>PoE</li></ul> | Power from the connector that the user has likely gone to most effort to supply power over
| POW-08 | The GTE shall provide a 0.75A maximum steady-state at a nominal 12V derived from whatever supply has been selected to power the device to the instrument-specific hardware. | So the instrument-specific hardware has a standard power supply. | 
| POW-09 | The DC voltage on the 12V rail must be within the range 10.5V - 16V for all steady state currents from no-load to maximum steady-state current | Allows use of 13.8V supplies and running from a vehicle supply (limits from BS EN 7637 from memory) | 
| POW-10 | The average power consumption of the GTE must not exceed 3W (averaged over any 10s window during any operational state) | To keep the total power consumption below the 15W limit of 802.3af |


# User interface
| Requirement ID | Requirement | Justification |
| --- | --- | --- |
| UX-01 | GTE must provide a generic, flexible user interface | So the UI can be configured to the specific instrument that the GTE is being used to create without new hardware |
| UX-02 | The unit must visually indicate to the user the distinction between: <ul><li>USB VBUS not present</li><li>USB PD unable to negotiate acceptable configuration</li><li>USB PD has negotiated acceptable configuration</li></ul> | If USB-PD is unable to reach a successful configuration, this must be indicated to the user rather than them thinking that the PSE is faulty or not supplying power |
| UX-03 | The unit must visually indicate to the user the distinction between: <ul><li>PoE voltage not present</li><li>PoE unable to negotiate acceptable configuration</li><li>PoE has negotiated acceptable configuration</li></ul> | If PoE is unable to reach a successful configuration, this must be indicated to the user rather than them thinking that the PSE is faulty or not supplying power |
